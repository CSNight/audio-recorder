## 背景：为什么整体重写

诊断脚本（`playground/stereo-diagnostic.html`）真机取证确认了三件事，推翻了 PR4 的 MediaRecorder 绕图实现：

1. **MediaRecorder 绕图是净亏损**。旧实现 `source → GainNode(explicit,N) → MediaStreamDestination → MediaRecorder` 仅为强制声道数而存在，实测：① “双声道”是假立体声（左右完全相同）；② 破坏浏览器原生 APM（AEC/NS/AGC）；③ 绕图后码流 `decodeAudioData` 解不开。
2. **声道数交给原生即可**。蓝牙 HFP 物理单声道，`channelCount:{exact:2}` 会诚实地 `OverconstrainedError`；有线/USB 立体声设备直接给真双声道。
3. **原生处理本就生效**，只要不绕图破坏它——三条采集路径消费同一条 `getUserMedia` 流。

目标：把 `src/input/` 整体重写为更干净的架构，零历史负债。支持 media-recorder / audio-worklet / script-processor 三模式，默认优先 MediaRecorder 按兼容性逐级降级，支持用户手动三选一，用浏览器原生约束提供增益/回声消除/降噪，如实上报实际采集链路。

## 清除的历史负债

- `InputGraph.inputNode`：MR 路径塞 `gain=0` 占位节点凑接口形状（仅测试在用）——删除。
- `createInputGraph` 巨型函数 + 三路径内联 + connect/disconnect 样板重复两遍——拆分为独立 backend。
- MR 路径 settled/succeed/fallback 手搓 Promise + 绕图节点两处创建——重写。
- `deactivateInputNode` 以节点命名的泄漏生命周期——换成语义化 `suspend/resume/dispose`。
- session 构造函数隐式 `connectGraph` 时序耦合——改为显式装配（先建 session 作为 sink，再注入 backend）。
- `runtimeInfo.inputStrategy` 上报能力预测值而非实际链路——改为如实上报 `backend.strategy`。

## 新架构：InputBackend 抽象

核心思路：三种采集 API 各实现一个统一的 `InputBackend`，由独立的选择/降级编排器决定用哪个；backend 只管把原始帧推给 sink，与状态机/转换解耦。

- **帧契约** `InputFrameSink.acceptFrame(planar, timestamp)`，由 `BrowserInputSession` 实现。
- **backend 接口**（`input/backends/types.ts`）：`strategy` / `suspend()` / `resume()` / `dispose()`。工厂统一签名 `(ctx) => Promise<InputBackend>`，不可用时抛 `InputBackendUnavailableError` 触发降级。
- backend 创建即“已接好流、开始产帧”（worklet/SP 图已连，MR 已 start 并通过 onstart 验证）；session 在非 recording 态由 sink 门控丢帧，三模式生命周期完全一致，无 inputNode 泄漏。

### 三个 backend（`input/backends/`）

- `media-recorder-backend.ts`：直接 `new MediaRecorder(stream, {mimeType:"audio/webm; codecs=pcm"})`，**无绕图**。`isTypeSupported` 预检 + `start(10ms)` + onstart/500ms 超时验证；`ondataavailable → webmExtract → sink.acceptFrame`；采样率不匹配发一次性 warning。`suspend/resume → mr.pause()/mr.resume()`。
- `audio-worklet-backend.ts`：`ensureWorkletRegistered`（`WeakMap<AudioWorklet>` 缓存）+ `new AudioWorkletNode`。图：`source → workletNode → sinkGain(0) → destination`，原生 APM 在 track 上。移动端 `batchSamples`（约 60Hz）保留，processor 源内联为常量模块。`suspend/resume` 空操作，由 session 门控。
- `script-processor-backend.ts`：降级兜底（deprecated）。图：`source → scriptProcessor → sinkGain(0) → destination`。

每个 backend 自带 connect/disconnect 私有逻辑，消除原先重复两遍的样板。

### 选择 / 降级编排器（`input/backends/select.ts`）

- 候选顺序：`"auto"` → `[media-recorder, audio-worklet, script-processor]`；显式值 → `[<显式值>, ...其余按标准优先级]`，保证显式不可用仍能降级。
- 依次 `await factory(context)`，捕获 `InputBackendUnavailableError` → 发降级 warning（MR 失败用 `MediaRecorderFallback`，worklet/SP 失败用 `ScriptProcessorFallback`）→ 试下一个。
- 返回首个成功者，其 `strategy` 即实际链路；最后一个候选失败不再发“降级”warning（无处可降），错误向上抛。

### 约束层（`input/audio-constraints.ts`，从 adapter 拆出）

- `buildAudioConstraints(input)`：三处理项默认 true；`channelCount` 用 `{ exact }`（仅显式传入时）；`deviceId` 用 `{ exact }`。
- `acquireMicStream(input, emitIssue)`：`getUserMedia`；捕获 `OverconstrainedError` → 发 `AudioConstraintNotApplied` warning → 去掉 exact 重试一次，`open()` 不硬失败。
- `reportUnappliedConstraints(stream, input, emitIssue)`：`track.getSettings()` 比对请求 vs 实际（三处理项 + channelCount），仅自有流执行。

### 采集会话（`input/browser-input-session.ts`）

`BrowserInputSession implements RecorderInputSession, InputFrameSink`：

- 持有 audioContext / stream / backend / 状态机；构造函数不再隐式连图，通过 `attachBackend()` 显式装配。
- 状态机 `ready → recording → paused/stopped → closed`：`start/resume → backend.resume() + audioContext.resume()`；`pause/stop → backend.suspend()`；`close → backend.dispose() + 停 track（自有流）+ 关 audioContext`。
- `acceptFrame`：非 recording 门控丢帧 → 丢帧补偿（vendor envInFix 算法，行为保留、实现重写干净）→ `createAudioFrame` 转 Int16 → `handlers.onFrame`。
- 暴露 `actualSampleRate` / `actualChannelCount`（取 `getSettings().channelCount` 真实值）/ 新增 `actualInputStrategy`（= `backend.strategy`）。

### 控制器（`input/browser-input-adapter.ts`）

`open()` 流程：acquireMicStream / 复用 sourceStream → 校验音轨 → 自有流跑 `reportUnappliedConstraints` → 建 AudioContext → 先 `new BrowserInputSession`（作为 sink）→ `selectInputBackend({requested: input.inputStrategy ?? "auto", context})` → `session.attachBackend(backend)` → 返回 session。全部 backend 失败时清理 audioContext 再抛。`listMicrophoneDevices` 留在本文件保持导出路径。

## 踩坑

- `channelCount:{exact:2}` 是关键。非 exact 时浏览器会静默降级到单声道（旧绕图实现正是为了规避它才引入绕图，反而引出假立体声）；改用 exact 后，蓝牙 HFP 会诚实抛 `OverconstrainedError`，再回退非 exact + 发 `AudioConstraintNotApplied` 警告，对用户透明。
- MediaRecorder backend 的可用性必须用 `onstart` 事件 + 500ms 超时验证：`isTypeSupported` 返回 true 但实际不产帧的情况在真机出现过，仅靠静态检查会假阳性。
- worklet `sinkGain(0) → destination` 不可省：部分浏览器在节点未接到 destination 时会暂停 `process()` 回调。增益 0 保证不产生可听输出。
- `BrowserInputSession` 显式装配解决了旧版构造期连图的时序耦合——session 必须先作为 sink 存在，backend 工厂才能在创建时就开始往 sink 推帧。

## 第二阶段：编码器 DI 化 + 多入口 Chunk 完全消除

### 背景

`EncoderRegistry` + `createDefaultEncoderRegistry()` 让 `index` 入口隐式 import 了 `encoders/pcm`、`encoders/wav`，间接拖入 `pcm-exporter`/`wav-header`/`resample`，与 `codecs/mp3` 等子入口共享模块，被 Rollup 提取成 `dist/chunks/*.js`。目标：编码器改为显式 DI 注入，彻底删除 `EncoderRegistry` 类与默认注册副作用，让每个 codecs 子入口自成闭包，不再与 `index` 共享底层模块。

### 架构变更

- **删除** `src/encoders/` 整个目录（`encoder-registry.ts`、`encoders/pcm/index.ts`、`encoders/wav/index.ts`），不再有 `EncoderRegistry` 类、`createDefaultEncoderRegistry()`。
- **新增** `SnapshotEncoderDefinition<TType, TOptions, TResult>` 类型，定义在 `src/types.ts`：`{ type: TType; export(snapshot, options?): TResult }`。
- **新增** `src/codecs/pcm/pcm-snapshot-encoder.ts`、`src/codecs/wav/wav-snapshot-encoder.ts`：分别导出 `pcmSnapshotEncoderDefinition`、`wavSnapshotEncoderDefinition`，薄封装各自的 `export*Snapshot` 函数。
- **`RecorderController`**：`encoderRegistry: EncoderRegistry` 字段删除，改为内部 `Map<string, SnapshotEncoderDefinition>`；构造函数新增可选 `encoders?: SnapshotEncoderDefinition[]` 参数，逐个 `set` 进 Map；不再有任何默认编码器。`registerEncoder()` 直接操作 Map；`exportEncoded(type)` 找不到注册项时抛 `Recorder encoder "${type}" is not registered. Pass it via createRecorder({ encoders: [...] }) or recorder.registerEncoder(...)`。
- **`createRecorder(options)`**（`src/index.ts`）：新增 `encoders` 透传，`new RecorderController({ ..., encoders: encoders ?? [] })`（因 `exactOptionalPropertyTypes`，用 `?? []` 避免把 `undefined` 传给非可选数组类型）。
- **`codecs/base` 合并入口**：新增 vite 入口 `"codecs/base/index"` 指向 `src/codecs/wav/index.ts`，该文件同时导出 PCM 和 WAV 的 chunked + snapshot 编码器（`pcmChunkedEncoderDefinition`、`pcmSnapshotEncoderDefinition`、`wavChunkedEncoderDefinition`、`wavSnapshotEncoderDefinition`），不再单独保留 `codecs/pcm/index` 和 `codecs/wav/index` 作为独立入口——只用一个组合入口承载两者，避免 PCM 相关模块被两个入口（pcm 独立入口 + wav 内部依赖 pcm 工具）同时引用而提取 chunk。
- **`vite.config.ts`**：`rolldownOptions.external: ["@scope/audio-recorder"]`，保证各子入口之间不会因为互相引用主包模块而被重新打包到一起；`storage/opfs`、`storage/indexeddb` 两个入口原先各自 import `src/utils/snapshot-codec.ts` 造成共享 chunk，现改为从主包（`@scope/audio-recorder`，external）取用，因为 `index.ts` 已经导出了 `serializePcmSnapshot`/`deserializePcmSnapshot`。

### 用户调用方式变更（Breaking Change）

```ts
// 旧：默认自带 pcm/wav 编码器
const recorder = createRecorder()
await recorder.exportEncoded("pcm")

// 新：必须显式传入
import { pcmSnapshotEncoderDefinition, wavSnapshotEncoderDefinition } from "@scope/audio-recorder/codecs/base"
const recorder = createRecorder({
  encoders: [pcmSnapshotEncoderDefinition, wavSnapshotEncoderDefinition],
})
await recorder.exportEncoded("pcm")

// 或运行时动态注册：
recorder.registerEncoder(wavSnapshotEncoderDefinition)
```
