## Phase 2 持久化溢写接口化

- 目标：
  - 解决浏览器长录音全量 PCM 常驻内存的问题，但不把 OPFS / IndexedDB 写死进核心默认路径。
- 完成：
  - 新增 `RecorderPersistencePlugin`、`RecorderStorageOptions` 等核心存储接口，只在主库内保留接口与 session 生命周期管理。
  - 新增 `SpillablePcmBufferStore`，支持 `memoryThresholdBytes` 超限后从 memory 切到持久化后端。
  - 持久化模型按“历史快照一次落盘 + 后续小批次持续写入”收敛，避免 spill 后仍持续保留全量内存。
  - 新增 OPFS 与 IndexedDB 可选实现模块；只有显式引入对应模块时，才会接入该能力。
  - 持久化 session 在 `clear()/close()/destroy()` 时执行清理，并在新 session 建立时先清理旧残留，避免浏览器缓存越积越多。
  - `RecorderController.exportPCM()/exportWAV()` 收敛为异步导出，以兼容持久化快照读取。
  - 新增 snapshot codec 与 spill store 单测，覆盖基础序列化和 session 级清理语义。

## Phase 2 Chunk 聚合内核抽象

- 目标：
  - 评估“memory 是否可以也做成类似 persist 的增量 chunk 组织，从而抽出统一抽象层”是否可行，并在不牺牲连续导出性能的前提下落地。
- 结论：
  - 不能直接把 `memory` 降成“每次读取都 drain 的增量 store”，否则连续 `snapshot()/export` 会重新全量 merge，性能会倒退。
  - 可行做法是抽出共享的 chunk 聚合内核，同时保留两种读取语义：
    - `snapshot`：给 `InMemoryPcmBufferStore` 用，保留 merged snapshot 缓存
    - `drainSnapshot`：给持久化后的 pending spill batch 用，读取后立即清空
- 完成：
  - 抽象过一轮共享 chunk 聚合逻辑，用于验证 memory / persist 是否能共用底层数据组织。
  - 最终保留的结论是：可以复用思路，但不值得单独保留一个跨文件 helper，后续已回收进具体 store 内部实现。
  - 新增单测覆盖：调用方修改第一次 `snapshot()` 返回值后，不能污染第二次导出结果，确保内存快照缓存仍对外隔离。
- 变更文件：
  - `src/buffer/in-memory-pcm-buffer-store.ts`
  - `src/buffer/spillable-pcm-buffer-store.ts`
  - `tests/unit/pcm-buffer-store.spec.ts`
  - `docs/architecture/execution-chain.md`

## Phase 2 Buffer Store 按方案整体重写

- 目标：
  - 停止在已有中间态上继续修补，按一套先明确语义再实现的方案整体重写 buffer-store，消除 `persistent` 和 `complex` 职责交叉、激活前缓冲混入 persistent 模式、重复 layout 校验分支等问题。
- 最终方案：
  - `InMemoryPcmBufferStore`
    - 纯同步内存缓冲
    - 负责 layout 校验、planar chunk 累积、snapshot merge、drain 和 clear
  - `PersistPcmBufferStore`
    - 纯持久化 store
    - 必须在 `open()` 阶段 `initialize()` 成功后才能开始录音
    - 不再承担任何激活前缓冲或 memory fallback
  - `ComplexPcmBufferStore`
    - auto 模式唯一实现
    - 负责 `memory -> promoting -> persist` 三阶段流转
    - promotion 期间用独立过渡缓冲接住新帧；promotion 失败时并回 memory
  - `createPcmBufferStore()`
    - 只做模式分派，不夹带额外策略
- 完成：
  - `InMemoryPcmBufferStore` 重写为单一 layout 校验入口 `ensureLayout()`，不再保留重复的 `establishFrameLayoutOrRejectMismatch / establishSnapshotLayoutOrRejectMismatch`。
  - `PersistPcmBufferStore` 重写为真正的“预激活后录音”模型：`initialize()` 失败直接让 `open()` 失败；`appendFrame()` 在未初始化时直接抛错。
  - `ComplexPcmBufferStore` 重写为显式三阶段状态机，并把 promotion 期间新增 frame 单独缓冲，避免切换窗口语义含混。
  - `PcmBufferStore` / `RecorderFramePipeline` 保留可选 `initialize()` 钩子，`RecorderController.open()` 在 capture adapter 打开前先初始化 pipeline。
  - 单测同步对齐：`persistent` 模式验证初始化失败；`complex` 模式验证自动提升与切换后清理。
